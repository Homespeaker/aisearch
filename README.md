# LocalDocSearch

Локальный поисковый движок на C++ для папок с документами.  
Проект индексирует `TXT`, `Markdown`, `DOCX` и `PDF`, строит обратный индекс и ищет релевантные документы и фрагменты без облачных API.

## Что умеет

- рекурсивно сканировать папку с документами;
- извлекать текст из `txt`, `md`, `docx`, `pdf`;
- разбивать документы на фрагменты;
- строить обратный индекс;
- искать по BM25;
- нормализовать слова;
- расширять запрос простыми смысловыми группами;
- делать fuzzy search;
- искать по имени файла и пути, даже если PDF не удалось прочитать;
- показывать фрагменты с подсветкой совпадений;
- работать полностью локально.

## Текущее состояние проекта

В проект уже добавлены:

- многопоточная индексация;
- кэш подготовленных документов в `.docsearch_cache`;
- сегментный индекс;
- двухступенчатый поиск `document -> fragment`;
- ускоренная обработка больших PDF с быстрым профилированием;
- защита от проблемных PDF без текстового слоя;
- поддержка путей с кириллицей и пробелами на Windows.

## Поддерживаемые форматы

- `.txt`
- `.md`, `.markdown`
- `.docx`
- `.pdf`

## Сборка

### MinGW

```powershell
cmake --fresh -S . -B build -G "MinGW Makefiles"
cmake --build build
```

### Visual Studio

```powershell
cmake -S . -B build
cmake --build build --config Release
```

После сборки бинарник находится здесь:

```powershell
.\build\docsearch.exe
```

## Быстрый запуск

Построить индекс:

```powershell
.\build\docsearch.exe build .\examples\docs -o .\docs_v4.idx -j 6
```

Искать по готовому индексу:

```powershell
.\build\docsearch.exe search .\docs_v4.idx "кто отвечает за оплату" -n 5 --answer
```

Построить временный индекс и сразу выполнить запрос:

```powershell
.\build\docsearch.exe query .\examples\docs "семантический поиск по pdf" -n 5 --answer -j 6
```

Интерактивный режим:

```powershell
.\build\docsearch.exe interactive .\docs_v4.idx --answer
```

Выход из интерактивного режима:

```text
:q
```

## Команды

```text
docsearch build <folder> [-o index.idx] [-j threads]
docsearch search <index.idx> <query words...> [-n limit] [--answer] [--no-color]
docsearch query <folder> <query words...> [-n limit] [--answer] [--no-color] [-j threads]
docsearch interactive <index.idx> [-n limit] [--answer] [--no-color]
```

## Как устроен поиск

1. Документы сканируются и приводятся к тексту.
2. Текст делится на фрагменты.
3. Для слов строится обратный индекс.
4. Запрос нормализуется так же, как и документы.
5. К запросу добавляются близкие термины и fuzzy-кандидаты.
6. Сначала ранжируются документы.
7. Затем ранжируются фрагменты только внутри лучших документов.

Это снижает стоимость поиска на больших корпусах по сравнению с прямым ранжированием всех фрагментов сразу.

## Кэш и повторные сборки

При повторной индексации проект использует локальный кэш:

```text
.\.docsearch_cache\<root-hash>\
```

Если файл не изменился по пути, размеру и времени модификации, повторного извлечения текста не происходит.

## PDF и DOCX

### DOCX

`DOCX` извлекается локально через ZIP-структуру файла и XML `word/document.xml`.

### PDF

Для PDF проект использует несколько уровней извлечения:

1. `pdftotext` из Poppler;
2. `mutool` из MuPDF;
3. встроенный простой fallback для небольших файлов.

Если PDF состоит из сканов или изображений без текстового слоя, файл все равно попадает в индекс по:

- имени файла;
- названию;
- пути;
- типу файла.

### Оптимизация PDF

Перед запуском тяжелых извлекателей проект делает быстрый профиль PDF:

- размер файла;
- наличие текстовых PDF-маркеров;
- признаки image-only/scanned PDF.

Для больших и подозрительных PDF:

- таймауты уменьшаются;
- лишние извлекатели не запускаются;
- fallback-парсер не включается без явных признаков текста.

Это заметно ускоряет обход больших библиотек.

## Зависимости

Обязательных внешних онлайн-зависимостей нет, но для лучшего PDF-поиска полезны:

- `pdftotext`
- `mutool`

Если их нет, программа продолжит работать, но часть PDF будет читаться хуже или попадет в индекс только по имени файла.

## Ограничения

- это не LLM и не генеративный ответчик; режим `--answer` собирает извлекающий ответ из найденных фрагментов;
- качество поиска по PDF сильно зависит от наличия текстового слоя;
- очень большие старые индексы лучше пересобирать новой версией бинарника, чтобы получить сегменты и актуальные оптимизации.
